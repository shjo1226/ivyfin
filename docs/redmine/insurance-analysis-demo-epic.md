# [Epic] 보험 보장분석 서비스 데모 고도화

작성일: 2026-03-18  
상태: 진행 중

## 1. 개요

### 목적
보험 점검 및 보장 분석에 관심 있는 고객을 대상으로, 채팅 상담과 음성 통화 상담을 통해 초기 안내를 자동화하고 방문 상담 예약까지 연결하는 데모 서비스를 고도화한다.

### 핵심 목표
- 채팅과 음성 통화에서 실제 고객 응대 흐름에 가까운 상담 플로우를 구현한다.
- GA 전문가 상담 전, AI가 고객 기본 정보, 건강 상태, 보험 현황, 방문 상담 정보를 자동 수집하고 정리한다.
- 수집된 상담 데이터를 구조화하여 `ConsultationRecord`에 저장하고, 고객용/관리자용 화면에서 쉽게 확인할 수 있게 한다.
- 방문 상담 예약 내역을 관리자 화면에서 표 형태로 조회하고 파일로 내려받을 수 있게 한다.

## 2. 현재 서비스 기준 범위

### 현재 반영된 범위
- 채팅 상담 프롬프트 분리 및 하이브리드 응답 UX 적용
- 음성 상담 프롬프트 분리 및 Gemini Live 기반 실시간 통화 처리
- 채팅/음성 공통 구조화 데이터 저장 테이블 `ConsultationRecord` 도입
- 상담 상태 관리: `collecting`, `interrupted`, `submitted`
- 고객용 상담 내역 페이지 `/records`
- 관리자용 방문 상담 예약 현황 페이지 `/admin/consultations`
- 관리자용 CSV 다운로드 API 제공

### 이번 Epic에서 유지하는 원칙
- 채팅과 음성은 수집 방식은 다르지만 저장 목표 데이터는 최대한 동일하게 유지한다.
- 방문 상담은 보장 분석과 신규 보험 상담 모두의 핵심 후속 단계로 본다.
- 희망 방문 날짜는 현재 서비스 정책상 선택 정보로 유지한다.
  이유: 실제 상담 흐름에서 고객이 먼저 날짜를 말할 때만 참고용으로 저장하도록 설계되어 있음.

## 4. 상담 데이터 정의

현재 프로젝트의 실제 저장 구조를 기준으로 정리한다.

### Part 1. 고객 기본 정보

| 필드명 | 현재 구현 필드 | 설명 | 필수 여부 |
| --- | --- | --- | --- |
| name | `customerName` | 고객명 | 필수 |
| phone | `customerPhone` | 연락처 | 필수 |
| birth_date | `customerBirthDate` | 생년월일 | 필수 |
| birth_date_confirmed | `birthDateConfirmed` | 고객이 생년월일을 확인했는지 여부 | 권장 |
| customer_confirmed | `customerConfirmed` | 고객 본인 확인 여부 | 권장 |

### Part 2. 건강 및 보험 상태

| 필드명 | 현재 구현 필드 | 설명 | 필수 여부 |
| --- | --- | --- | --- |
| recent_medical_history | `recentMedicalHistory` | 최근 입원, 수술, 암/뇌혈관/심혈관 진단 이력 | 권장 |
| current_medication | `currentMedicationStatus` | 정기 복용 약 여부 | 권장 |
| current_insurance | `currentInsuranceSummary` | 현재 유지 중인 보험 요약 | 권장 |
| monthly_premium | `monthlyPremium` | 월 보험료 규모 | 선택 |
| interest_areas | `interestAreas` | 관심 보장 분야 및 상담 의도 | 권장 |

### Part 3. 방문 상담 예약 정보

| 필드명 | 현재 구현 필드 | 설명 | 필수 여부 |
| --- | --- | --- | --- |
| visit_region | `visitRegion` | 선호 방문 지역 | 필수 |
| visit_address | `visitAddress` | 상세 방문 주소 | 권장 |
| address_type | `visitAddressType` | 자택, 직장, 카페 등 방문 장소 유형 | 권장 |
| preferred_date_year | `preferredVisitYear` | 희망 방문 연도 | 선택 |
| preferred_date_month | `preferredVisitMonth` | 희망 방문 월 | 선택 |
| preferred_date_day | `preferredVisitDay` | 희망 방문 일 | 선택 |
| preferred_time_period | `preferredVisitTimePeriod` | 오전, 오후 | 선택 |

| notes | `notes` | 기타 요청 사항 | 선택 |

### 데이터 정책 메모
- `preferred_date`를 단일 필수 필드로 두지 않는다.
- 현재 서비스는 고객이 주소만 남겨도 접수 가능하며, 날짜와 시간은 고객이 먼저 말했을 때만 참고용으로 저장한다.
- 이 정책은 현재 음성/채팅 프롬프트와 운영 흐름에 맞으므로 유지한다.

## 5. 채팅 vs 음성 수집 방식

| 구분 | 채팅 상담 | 음성 통화 상담 |
| --- | --- | --- |
| 진입 방식 | 홈페이지에서 고객이 직접 진입 | 관리자가 연결한 아웃바운드 통화 |
| 기본 전제 | 개인정보 수집 동의 후 자유 상담 | 신청 정보 일부가 이미 있을 수 있음 |
| 장점 | 이름, 생년월일, 주소 등 정형 정보 확인이 쉬움 | 건강 상태, 보험 니즈, 반론 대응 같은 정성 정보 확보에 유리 |
| 현재 강점 | 스트리밍/분석 하이브리드 UX, 본인 확인 흐름 안정화 | Gemini VAD 인터럽트, 실시간 대화 로그, 텍스트 테스트 입력 지원 |
| 운영 포인트 | 질문 단위를 짧게 유지하고 방문 상담 접수까지 연결 | 불필요한 재확인 질문을 줄이고 빠른 템포로 진행 |

## 6. 현재 구현 기준 변경 요약

### 서비스 기준으로 확정한 내용
- 보험 보장 분석과 신규 보험 상담 모두 방문 상담 접수까지 연결한다.
- 채팅 상담 고객은 이미 개인정보 수집을 마친 상태를 전제로 한다.
- 음성 상담은 신청 정보가 일부 있는 고객에게 연결되는 아웃바운드 상담을 전제로 한다.
- 상담 종료 전에는 반드시 마무리 멘트가 먼저 나가고, 그 뒤 `end_call` 흐름으로 들어간다.
- 고객이 종료 직전 다시 말하면 종료를 취소하고 후속 응대를 이어간다.

### 문서 초안 대비 조정한 내용
- "보험 보장분석 커미션 안내" 표현은 현재 서비스 맥락과 맞지 않아 제거한다.
- `preferred_date`, `preferred_time_period`를 필수로 두는 안은 현재 운영 플로우와 충돌하므로 선택 정보로 조정한다.
- 고객용 `/records`와 관리자용 `/admin/consultations`를 역할별로 분리한다.
- 관리자 페이지에서는 전체 방문 상담 예약을 표 형식으로 조회하고 CSV로 내려받을 수 있다.

## 7. 남은 우선 과제

### 우선순위 높음
- 채팅 상담에서 버튼형 선택지, 주소 입력 보조, 예약 보조 UI 추가
- 음성 상담의 단계별 프롬프트 안정화와 조기 `end_call` 방지
- `ConsultationRecord` 추출 정확도 개선

### 우선순위 중간
- 고객용 상담 상세 페이지 접근 제어 고도화
- 관리자 대시보드 필터, 상태 변경, 메모 기능 추가
- CSV 외 실제 XLSX 다운로드가 필요하면 파일 포맷 확장 검토

## 8. 완료 기준

- 채팅과 음성 상담 모두 방문 상담 접수 정보가 구조화 저장된다.
- 고객은 본인 상담 내역만 `/records`에서 확인할 수 있다.
- 관리자는 `/admin/consultations`에서 전체 예약을 표 형태로 확인할 수 있다.
- 관리자는 전체 예약 내역을 CSV 파일로 다운로드할 수 있다.
- 상담 종료 전후 흐름에서 불필요한 질문과 중복 마무리 멘트가 최소화된다.
